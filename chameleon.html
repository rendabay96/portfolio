Chameleon

A multivariate testing tool for internal teams to configure tests for our partners.

Project Overview

Our internal teams have had a notoriously rocky relationship with our internal ad configuration and multivariate testing tool called Chameleon. In order to allow our users to work more scalably as more partners opted into our media applications, we embarked upon a complete redesign to address major user needs that had built up over time. As lead designer I owned research and design, working closely with engineers to modernize the system.

User Research

To kick off discovery for this long-awaited redesign, I interviewed 12 primary users of Chameleon. This included partner integration engineers, PMs, front-end dev teams, and optimization managers - each with their own level of expertise in launching and monitoring multivariate tests (as well as navigating the tool’s shortcomings).

After learning their processes, shadowing their daily workflows, and documenting their frustrations, I imported all raw interview notes into a research repository and developed a qualitative data tagging system.

Dovetail, a UX Research repository tool I used to spot trends in pain points such as “inefficiency” and “dependency.”


Two of the main insights we synthesized to guide our feature recommendations were:
  1. The Search functionality has to be more robust to return results that users actually care about.
  2. To reduce errors caused by lack of UI feedback, we should address UX improvements around configuration creation.

The Objective

With the increasing volume of multivariate tests that we run for our partners, our internal teams struggled to work scalably due to Chameleon’s shortcomings when it came to their test-launching workflows, particularly around the Search feature as well as Configuration Creation. Our goal was to redesign Chameleon to help remove pain points and dependencies preventing our teams from efficiently launching and maintaining thousands of ongoing tests.

Designing The Experience

Redesigning Chameleon inevitably involved rebuilding a lot of the functionality that exists today, so to get buy-in from our stakeholders (the team leads and PM) I mapped out the primary user flows along with feature recommendations informed by our user insights above. We used Figma to share the recommendations in the context of the user flow so that the engineers can begin prioritizing.

User Flow 1: Search Configuration

Our user research showed that the existing search results page had many shortcomings that slowed our users down when they were searching for specific configurations.

Original Search Page

Frequently mentioned user paint points around the Search functionality were as follows:

  1. 100% of users expressed the need for more filters. Majority were frustrated with their lack of text matches at a more granular level.
  2. There's a limited number of results in view at a time, especially when expanding cards for context.
  3. Majority of users need additional sort features. The learning curve for certain sort metrics also created dependencies.


Redesigned Search page

Redesigned Search Page

A few iterations and user testing sessions later, our solution for the filters, list results, and sorting were implemented:
  1. Exclusions, ability to search within JSON strings, and ability to add filters sped up filtering.
  2. A table format makes scanning for information much easier; more configurations are within view.
  3. Additional helpful sort-by features and tooltips on hover for context were added.

Building + Testing

To build our MVP our engineer and I used the Vuetify framework with ready-made Material Design components. Soon we were able to ship a functioning Search MVP for Chameleon 2.0 and test for more user feedback.

Chameleon 2.0 - Search



So far the new Search’s positive reception has been validating the new design. It's great to know that we’ve delivered a much more robust Search feature handling thousands of configurations serving custom treatments for our partners sites. These are some of the direct quotes from our key users on how they're adapting to the new Search design:
  • “It’s definitely faster.”

  • It lived up to expectations “based on the mocks we saw.”

  • “This has been a pretty painless transition” so far.

User Flow 2 (preview): Create New Configuration

The main insight from user research informing the solution for this portion of the tool was that we need to reduce human error and inefficiencies caused by lack of UI feedback. One major UX improvement around configuration creation we wanted to focus on was that of assigning variants. This is not simply an A/B testing tool we’re building. Often we test 3+ variants for our partners, so theoretically, the UI must accommodate any number of variants.

Original Variants Page

Here's an example of how overwhelming the current flow for setting up test variants can be. After a series of steps, the user reviews a huge matrix (requiring scrolling in both directions) just to select 4 specific variants she wants to activate for the test.

Original Variants Page

Redesigned Variants Page

We reimagined a possible flow that would reduce cognitive load. Despite essentially reversing the original process in this new UI, majority of our users have praised its intuitivenss and the removal of unnecessary steps to achieving their user goal.

Redesigned Variants Page - Prototype

Next Steps

We’re in the midst of iterating on several new user flows to reach our objective in simplifying the overall process of setting up a multivariate test. So far, the user feedback for Use Flow #1: Search has been a combination of positive as well as requests for even more capabilities now that the users have had time to experiement and get a sense of its potential!

On User Flow #2: Create Configurations- we'll continue to test the prototype and iterate as necessary. In general we're proud of the progress we've made thus far, and I’m looking forward to analyzing more feedback so we can build out and iterate on the rest of the feature recommendations.
